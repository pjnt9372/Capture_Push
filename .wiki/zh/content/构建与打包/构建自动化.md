## 简介
构建自动化是确保 Capture_Push 项目质量和发布效率的关键。本项目主要利用 **GitHub Actions** 实现持续集成和持续部署 (CI/CD)，同时支持开发者在本地通过脚本快速验证构建。

## CI/CD 概览
自动化流程涵盖以下主要环节：

1.  **代码检查**：代码风格检查和静态分析。
2.  **单元测试**：运行 Python 单元测试，确保逻辑正确。
3.  **构建托盘**：编译 C++ 托盘程序。
4.  **打包发布**：生成 Inno Setup 安装包并发布到 GitHub Releases。

## GitHub Actions 工作流
项目定义了标准的工作流文件（通常位于 `.github/workflows/`），主要包含：

### 触发条件
- `push` 到 `main` 分支。
- 创建以 `v` 开头的标签（如 `v1.0.0`），触发发布流程。
- `pull_request` 事件。

### 构建步骤
1.  **环境设置**：
    - 检出代码 (`checkout`)。
    - 设置 Python 环境 (`setup-python`)。
    - 设置 MSVC 编译器（用于 C++）。
    - 安装 Inno Setup 编译器。

2.  **依赖安装**：
    - `pip install -r requirements.txt`

3.  **编译与打包**：
    - 运行 CMake 构建托盘程序。
    - 运行 `ISCC` 编译 `.iss` 脚本。

4.  **产物上传**：
    - 将生成的 `.exe` 安装包上传为 Artifacts 或 Release Assets。

## 本地自动化
虽然 `developer_tools/build.py` 已不再作为主要构建入口，但开发者仍可以通过编写简单的批处理脚本 (`.bat` 或 `.ps1`) 来串联构建步骤，模拟 CI 环境。

示例流程：
```powershell
# 1. 安装依赖
pip install -r requirements.txt

# 2. 编译托盘
cd tray
cmake -B build
cmake --build build --config Release
cd ..

# 3. 打包
ISCC Capture_Push_Setup.iss
```

## 结论
通过 GitHub Actions 实现的构建自动化，确保了每次代码提交都能通过测试并生成可用的安装包。这种“基础设施即代码”的方法大大降低了人为错误的风险，并加速了软件交付周期。
